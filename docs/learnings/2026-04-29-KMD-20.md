## 改善点

- **MVVM 境界違反**: `TemplatePickerView.onAppear` で `FileService()` を直接インスタンス化。View が Service を直接呼ぶパターンが新規ファイルに持ち込まれた
- **同期 I/O on MainActor**: `onAppear` 内の `loadTemplates()` がメインスレッドをブロックする実装になった。カスタムテンプレートが増えた際のシートオープン遅延リスクが残存
- **テストの空振り**: `ensureCustomTemplateDirectory` のテストが `FileService` を呼ばず `FileManager` を直接呼んでおり、実装ロジックを一切検証していなかった。テスト名と実装が乖離
- **ID 衝突リスク**: `DocumentTemplate.id` がファイル名のみで構成されており、ビルトインとカスタムで同名ファイルが存在すると SwiftUI の `ForEach` で不定動作が起きうる設計になった
- **PRD のカウント曖昧さ**: 「ビルトインテンプレート5種」と記載しつつ、「空ファイル」がファイルとして実装されるかUI選択肢かが不明確で、実装が4ファイルになった。レビューで指摘されるまで気づかれなかった

## 教訓

- **影響範囲マップの明示は実装精度を高める**: PRD に「変更してはいけないファイル」を列挙しておくことで、Codex エージェントのスコープ逸脱を防げる。今後も PRD テンプレートにこの項目を必須化すること
- **新規 View ファイルこそ MVVM チェックが必要**: 既存ファイルの慣習（SettingsView の ConfluenceService 直呼び）があっても、新規追加ファイルは正しいアーキテクチャで書く機会。実装プロンプトに「View から Service を直接呼ばず ViewModel 経由にすること」を明記することで防げる
- **onAppear の非同期化はデフォルトにすべき**: ファイルI/Oを伴う処理を `onAppear` に書く場合は `Task { await ... }` パターンをデフォルトとして採用すること。実装プロンプトに「ファイルI/Oは非同期で実装すること」と記載する
- **テストは実装対象を必ず経由させる**: テスト名に登場するメソッド（`ensureCustomTemplateDirectory`）はテスト内で必ず呼び出すこと。「テストが pass した」だけでなく「テストが意図した実装を検証している」かを確認する観点を実装プロンプトに追加する
- **ID 設計は衝突を前提にする**: SwiftUI の `Identifiable` を実装する際は、ユーザー入力（ファイル名）のみを id にしない。`"builtin:\(filename)"` / `"custom:\(filename)"` のような名前空間付き id をデフォルトとして採用する
