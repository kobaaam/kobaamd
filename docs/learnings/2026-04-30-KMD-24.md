## 改善点
- **README.md に KMD-17 の変更が混入**: PR diff に Diff viewer (KMD-17) 関連の README 変更が4箇所含まれていた。「1タスク=1PR」原則に違反。レビューで concern として指摘されたが、修正されずにマージされた
- **Linear MCP の truncation 問題**: レビュー時に PRD 全文を取得できず、section 8「影響範囲マップ」の検証が不可能だった。2つのレビューが同じ concern を独立して指摘しており、reviewer が PRD と照合できない構造的問題
- **concern がマージをブロックしなかった**: 3件のレビューが全て COMMENTED（concern あり）だったが、REQUEST_CHANGES に昇格せずそのままマージされた。concern の中には「README.md の KMD-17 混入」という明確な修正すべき指摘も含まれていた
- **テスト未追加**: HelpSection enum や View に対するテストがなし。静的 UI のため severity は低いが、`HelpSection.allCases` の網羅性テスト等は追加可能だった

## 教訓
- **concern と REQUEST_CHANGES の判定基準を明確化すべき**: 「別チケットの変更が混入」は明らかに REQUEST_CHANGES 相当だが、concern で留まった。レビュープロンプトに「無関係な変更の混入は fail」というルールが不足している
- **PRD-lite の長さ制限を意識する**: Linear MCP の truncation は PRD が長い場合に発生する。PRD-lite は「レビュー時に全文取得可能な長さ」に収める、または `docs/prd/` にファイルとして配置して Git 経由で読めるようにする運用が望ましい
- **README 更新は専用コミットまたは専用 PR で行う**: 実装 PR に README 更新を含めると、別チケットの変更が混入するリスクが高い。特に複数のチケットが近接して完了する場合に顕著
